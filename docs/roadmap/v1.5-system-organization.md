# Creation Station v1.5 - System Organization and Task-Board Scalability

## Goal

Plan the next stable milestone after `v1.1.0`.

v1.5 should make the existing local-first solo creator workflow easier to manage as the number of ideas, plans, and tasks grows. It should improve organization, scanning, filtering, and task-board scale without adding new AI features or changing the core workflow.

Baseline:
- Stable tag: `v1.1.0`
- Planning branch: `planning/v1.5-system-organization`
- Core workflow remains: Idea Inbox -> AI Factory Planner -> Review Inbox -> Revision -> Approval -> Tasks

## Scope

Allowed v1.5 work:
- Task Board scalability.
- Grouping and collapsible task sections.
- Active, backlog, and archived task separation.
- Labels, tags, and priority organization.
- Idea Inbox organization improvements.
- Small UI/UX improvements that preserve the current visual identity.
- Lightweight documentation and QA coverage.

v1.5 is a product-organization milestone, not an AI architecture milestone.

## Task Board Scalability

Problem:
- The current board is readable with a modest task count, but it will become harder to scan as approved plans create more tasks.
- A flat TODO-heavy board makes it difficult to separate immediate work from future work.

Planning direction:
- Keep the board compact and operational.
- Prefer sections, filters, and counts over large card redesigns.
- Preserve existing task records and task status behavior.
- Make the default view show what needs attention now.

Candidate improvements:
- Add an "Active" working area for tasks that should be handled soon.
- Add a "Backlog" area for deferred tasks.
- Add an "Archived" or hidden-completed view if completed or parked tasks begin to crowd the board.
- Add board-level counts for active, blocked, backlog, and done tasks.
- Keep empty states clear for each group.

Acceptance criteria:
- A user can scan many tasks without reading every card.
- TODO tasks are not presented as one long unstructured list.
- Blocked and high-priority work remains visible.
- Completed or deferred work does not dominate the default board.

## Grouping And Collapsible Sections

Current direction:
- v1.1 introduced task grouping and collapsible sections for TODO readability.

v1.5 should refine this pattern:
- Keep collapsible sections for groups with real tasks.
- Avoid showing empty groups unless they help orientation.
- Use section counts so collapsed groups remain informative.
- Keep default-open behavior focused on active work.
- Make collapsed state visually obvious without adding heavy UI.

Possible group dimensions:
- By task status: TODO, DOING, BLOCKED, DONE.
- By source plan or idea.
- By priority.
- By asset/setup/production/review type, if reliably derivable from existing data.
- By active/backlog/archive layer.

Decision rule:
- Prefer grouping based on existing data first.
- Do not add new schema just to support a nicer grouping label unless the current data cannot support the workflow.

## Active, Backlog, And Archived Task Separation

Need:
- Tasks need a clearer operating layer beyond raw status columns.
- A solo creator needs to distinguish "now", "later", and "not in the way" without losing task history.

Definitions for planning:
- Active: tasks visible in the normal working board.
- Backlog: valid tasks that should not distract from current work.
- Archived: tasks or task groups hidden from the main view because they are completed, obsolete, or intentionally parked.

Important risk:
- The current `Task` model has `status` and `priority`, but no dedicated backlog or archived field.
- v1.5 should first test whether separation can be represented with existing fields, filters, and view logic.
- If a real persisted task layer is required, document the schema need before changing Prisma.

Acceptance criteria:
- Default board does not feel crowded after multiple approved plans.
- Backlog work is still recoverable.
- Archived work is not confused with deleted work.
- No task data is destroyed when changing views.

## Labels, Tags, And Priority Model

Current state:
- Ideas have `tags`, `category`, `priority`, and `potential`.
- Tasks have `priority`, but no task-level tags or labels.
- Plans contain required assets and next actions as text.

v1.5 planning questions:
- Can task labels be derived from source idea tags, source plan sections, or task title patterns?
- Should task priority remain a simple field, or does the UI need better priority filtering first?
- Are labels only a display/filter layer, or do they need persistence?

Preferred approach:
- Start with existing data.
- Display idea tags or plan context on task cards only when it improves scanability.
- Add filters for priority before adding new label storage.
- Treat new task-label persistence as a schema risk, not a default requirement.

Acceptance criteria:
- High-priority tasks are easy to find.
- Tasks remain tied to their source idea or plan.
- Labels do not become a new taxonomy-management system.
- The UI does not require users to maintain labels before the board is useful.

## Idea Inbox Organization Improvements

Need:
- The Idea Inbox should remain useful as the number of captured ideas grows.
- Search, filters, archive visibility, and status labels should support quick triage.

Candidate improvements:
- Improve filter grouping and layout for narrow cards and mobile widths.
- Add clearer status filter sections or counts.
- Make archived idea visibility predictable.
- Improve empty-result guidance for search/filter combinations.
- Consider compact idea cards for dense review.
- Preserve the existing create idea flow.

Acceptance criteria:
- The user can find ideas by title, raw text, tags, category, or status.
- Archived ideas stay out of the default working view.
- Search/filter controls do not overlap or collapse awkwardly.
- The inbox remains usable without a new navigation system.

## Data Model Risks

v1.5 should avoid Prisma schema changes unless required.

Known risks:
- Task active/backlog/archive separation may require persisted state if view-only filters are not enough.
- Task labels may require a new field or model if derived labels are not reliable.
- Task grouping by source idea may require more efficient query shaping as data grows.
- Adding enum values or new models could create migration and data-integrity risk.

Guardrails:
- Do not change `prisma/schema.prisma` during planning.
- Before any schema change, document:
  - the user problem,
  - why existing fields cannot solve it,
  - the proposed minimal field/model,
  - migration and rollback risk,
  - expected UI behavior.
- Preserve existing task data and workflow behavior.
- Do not split the app into a new subsystem.

## UI/UX Acceptance Criteria

The v1.5 UI should:
- Preserve the current Creation Station visual identity.
- Improve spacing, alignment, readability, and hierarchy with targeted changes.
- Avoid large redesigns or new UI frameworks.
- Keep cards compact enough for repeated daily use.
- Make the task board readable with many tasks.
- Make filters and collapsible sections predictable.
- Keep destructive or hiding actions explicit and reversible where possible.
- Avoid showing long unstructured task lists by default.
- Work at desktop and narrow browser widths without overlapping controls.

The user should be able to:
- See what needs attention now.
- Park lower-priority work without deleting it.
- Find work by status, priority, source idea, or simple text.
- Understand when a task is active, backlog, completed, or archived.

## Must Not Be Included In v1.5

Do not include:
- New AI features.
- New AI agents.
- Agent meetings.
- External connectors.
- Plugin systems.
- Full asset vault.
- ComfyUI integration.
- VSCode automation integration.
- Calendar or team systems.
- Authentication.
- Cloud sync.
- Payments.
- Deployment infrastructure.
- Marketplace features.
- Major database redesign.
- New automation engine.
- v2.0 product architecture work.

Stop and ask before implementation if a v1.5 idea requires:
- new persistent models,
- cross-app architecture changes,
- external services,
- autonomous execution,
- replacing the existing workflow.

Planning checklist:

- [ ] No application code changes before the v1.5 implementation plan is accepted.

- [ ] Preserve existing task data.
- [ ] Run `npm run lint`.
- [ ] Run `npm run build`.
- [ ] Smoke test localhost.
- [ ] Verify task board readability with many tasks.
- [ ] Verify active/backlog/archive behavior, if implemented.

- [ ] Full idea workflow still passes.

- [ ] No accidental v2.0 scope included.

- [ ] Human review confirms board organization is useful in normal work.
