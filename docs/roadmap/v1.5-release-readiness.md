# Creation Station v1.5 - Release Readiness

## Goal

Create the final v1.5 roadmap from the current stable system.

Creation Station already has the core local-first workflow in place:

```text
Idea Inbox -> AI Factory Planner -> Review Inbox -> Revision -> Approval -> Tasks
```

Core completed:
- idea inbox
- planning
- review
- task generation
- task board
- intelligence layer
- dependencies
- modular pipelines
- export system

v1.5 should focus only on:
- onboarding
- UX polish
- reliability
- optional cloud persistence
- analytics
- release readiness

Do not introduce unnecessary architecture. Preserve the current local-first, solo-creator workflow and avoid changes that turn v1.5 into a v2 architecture milestone.

## A. v1.5 Milestone Breakdown

### v1.5.0 - Scope Lock And Baseline

Goal: make the active v1.5 direction explicit before more implementation work.

Included scope:
- document the release-readiness roadmap
- confirm the stable foundation from current v1.4.1 work
- preserve the previous system-organization roadmap as historical foundation context
- run baseline docs and repository checks before code work resumes

Excluded scope:
- app behavior changes
- Prisma schema changes
- branch deletion or destructive git cleanup

Acceptance criteria:
- `ROADMAP.md` points to this document as the active v1.5 plan.
- Older v1.5 organization planning remains available as foundation context.
- The next implementation slice is clear and small.

### v1.5.1 - Onboarding

Goal: make the first successful run obvious for a solo creator.

Included scope:
- improve first-use guidance for empty workspaces
- clarify local AI setup and Ollama readiness path
- guide the user from first idea to first approved tasks
- improve recovery tips when setup or planning fails

Excluded scope:
- account onboarding
- hosted setup flows
- multi-user onboarding
- new AI provider systems

Acceptance criteria:
- A new user can understand the first workflow without reading old release docs first.
- Empty-state onboarding points to Settings, Inbox, Factory, Review, and Tasks in the right order.
- Onboarding copy stays compact and does not become a tutorial subsystem.

### v1.5.2 - UX Polish

Goal: make the existing workflow easier to use repeatedly without redesigning the app.

Included scope:
- polish spacing, hierarchy, and button clarity
- improve Factory wait-state feedback
- reduce Review Inbox crowding
- clarify pipeline badges, task filters, and recommendation explanations
- verify narrow viewport layouts

Excluded scope:
- new UI framework
- major redesign
- new navigation system
- new workflow screens unless required for existing release readiness

Acceptance criteria:
- The app remains visually consistent with the current identity.
- Users can scan the Inbox, Review Inbox, and Task Board without reading every card.
- Controls do not overlap or collapse awkwardly on narrow viewports.

### v1.5.3 - Reliability

Goal: make the current core workflow boringly dependable.

Included scope:
- harden server-action validation
- protect approval and task creation from duplicate or stale operations
- verify task status transitions and dependency behavior
- re-check export integrity
- confirm large task-board rendering remains stable

Excluded scope:
- workflow rewrite
- new job system
- background worker
- automation engine

Acceptance criteria:
- The core workflow passes end to end.
- Duplicate clicks or stale review states do not create broken records.
- Export continues to include ideas, plans, tasks, and task blockers.
- Large local task sets do not create obvious browser or dev-server instability.

### v1.5.4 - Analytics

Goal: provide useful local insight from existing data.

Included scope:
- local-only dashboard improvements from existing Prisma data
- pipeline counts
- review bottlenecks
- stale tasks
- task status and priority summaries
- backup freshness or release-readiness signals

Excluded scope:
- external telemetry
- event tracking infrastructure
- productivity scoring
- historical analytics schema
- chart-heavy analytics system

Acceptance criteria:
- Analytics are derived from existing local database records.
- Counts match visible app data.
- Insights help the user decide what to review, continue, unblock, or back up next.

### v1.5.5 - Optional Cloud Persistence

Goal: make local backups easier to preserve without adding sync architecture.

Included scope:
- manual, opt-in backup persistence guidance
- export metadata clarity
- backup verification checks
- documentation for saving exports to a cloud-synced folder such as OneDrive, Google Drive, Dropbox, or iCloud Drive

Excluded scope:
- real cloud sync
- accounts
- authentication
- external connectors
- conflict resolution
- background upload/download
- cloud database
- import/restore unless explicitly approved as a separate slice

Acceptance criteria:
- Users can understand how to preserve local backups safely.
- The app does not require a cloud service.
- No new connector, auth, or sync subsystem is added.

### v1.5.6 - Release Readiness

Goal: prepare a stable v1.5 release candidate.

Included scope:
- final release checklist
- full workflow QA
- docs updates
- changelog and current-state updates
- final backup before release candidate
- release candidate tagging only after human approval

Excluded scope:
- any deferred v2 work
- last-minute feature expansion
- deployment infrastructure

Acceptance criteria:
- v1.5 has a clear QA record.
- No v2 scope accidentally ships in v1.5.
- Docs match the actual app behavior.
- The release can be resumed later from documented state, not memory.

## B. Implementation Order

1. Lock v1.5 scope in docs and confirm the stable QA baseline.
2. Polish onboarding first, because it clarifies the product without touching the core data flow.
3. Fix UX friction found during v1.4 and v1.4.1 QA: Factory progress feedback, Review Inbox crowding, pipeline visibility, recommendation explanation, and compact task scanning.
4. Harden reliability around server actions, task transitions, duplicate approvals, export shape, and large task boards.
5. Add analytics using existing Prisma data only.
6. Add optional persistence support as manual backup/export guidance and verification, not as cloud sync.
7. Run full release QA and update docs before tagging.

## C. Highest-Risk Areas

- Optional cloud persistence can easily become auth, sync, conflict resolution, account management, or connector work.
- Restore/import is higher risk than export and should not be added casually.
- Analytics accuracy is limited by existing `createdAt`, `updatedAt`, status, priority, task, plan, and idea data.
- Task board scale already exposed rendering pressure in earlier QA and should be rechecked with large local boards.
- Factory progress UX must not imply streaming, background jobs, or async worker systems that do not exist.
- Schema creep is likely around onboarding state, analytics history, cloud persistence, and labels; avoid it unless a specific user problem cannot be solved with existing data.

## D. What Should Wait For v2

- User accounts, authentication, permissions, and teams.
- Real cloud sync, background sync, merge/conflict handling, and multi-device state.
- External cloud provider APIs or connector marketplace.
- Event-sourced analytics, historical charts, productivity scoring, or telemetry infrastructure.
- Full restore/reconcile system for imported backups.
- Agent systems, automation runners, plugin systems, meetings, calendars, and asset vault expansion.
- Major Prisma redesign or new workflow subsystems.
- Deployment infrastructure and hosted product architecture.

## E. Launch Readiness Checklist

- [ ] `npm run lint` passes.
- [ ] `npx tsc --noEmit` passes.
- [ ] `npx prisma generate` passes if Prisma or generated client files are affected.
- [ ] `npm run build` passes.
- [ ] Localhost smoke test covers `/`, `/dashboard`, `/factory`, `/settings`, `/release`, and `/api/export`.
- [ ] Full workflow passes: Idea Inbox -> AI Factory Planner -> Review Inbox -> Revision -> Approval -> Tasks.
- [ ] Onboarding works for an empty database and does not appear when it is no longer useful.
- [ ] Factory loading and error states are understandable with Ollama online, offline, and missing-model cases.
- [ ] Task Board remains readable with many tasks, blockers, backlog, done, archived, and pipeline filters.
- [ ] Analytics match visible database counts and stay local-only.
- [ ] Export backup includes ideas, plans, tasks, and task blockers.
- [ ] Optional persistence is off by default and does not require accounts or external services.
- [ ] Docs are updated: roadmap, changelog, current state, QA report, release checklist.
- [ ] No v2 scope is accidentally included.
- [ ] Final backup is created before release candidate.
- [ ] Stable slices are committed with `Polish`, `Fix`, `Stabilize`, or `Document` messages.

## Explicit Deferrals

Do not add these in v1.5:
- authentication
- teams
- external connectors
- agents
- plugin systems
- full asset vault
- real cloud sync
- background jobs
- deployment infrastructure
- major architecture changes

If a v1.5 task appears to require one of these, stop and document the reason before implementation.
