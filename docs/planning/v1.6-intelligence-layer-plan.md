# Creation Station v1.6 Intelligence Layer Plan

Branch: `feature/v1.6-intelligence-layer`

Baseline: `v1.5.0` on `master`

Status: Planning only. No application code changes are included in this document.

## Current Baseline Summary

Creation Station v1.5 is a stable local-first solo creator workflow:

```text
Idea Inbox -> AI Factory Planner -> Review Inbox -> Revision -> Approval -> Tasks
```

The app is built with Next.js App Router, React server components, server actions, Prisma, and SQLite. The v1.5 release added task-board scalability, Idea Inbox polish, first-use onboarding, manual JSON backup/restore, and lightweight local analytics.

The current intelligence layer already exists as a small heuristic module in `src/lib/intelligence.ts`. It powers idea routing, task waiting state, task staleness, next-task ranking, and the home-page AI Recommendations panel. v1.6 should make this layer more coherent, testable, visible, and reliable without turning it into an agent system or a new architecture.

## Current Architecture Findings

### Framework

- Next.js `16.2.4` using the App Router under `src/app`.
- React `19.2.4`.
- TypeScript with server-first rendering.
- Styling is Tailwind-style utility classes in component markup.

### Routing Structure

- `/` in `src/app/page.tsx`: main Idea Inbox, Review Inbox, AI Recommendations, and Task Board surface.
- `/factory` in `src/app/factory/page.tsx`: AI Factory Planner.
- `/dashboard` in `src/app/dashboard/page.tsx`: database overview and status counts.
- `/settings` in `src/app/settings/page.tsx`: AI provider health, prompt presets, analytics, backup/restore.
- `/release` in `src/app/release/page.tsx`: release readiness page.
- `/api/export` in `src/app/api/export/route.ts`: manual workspace JSON export.
- `/api/analytics` in `src/app/api/analytics/route.ts`: limited client-side analytics event intake.

### State Management

- Server actions in `src/app/actions.ts` perform core mutations and call `revalidatePath`.
- URL query parameters drive Idea Inbox and Task Board filters.
- Browser `localStorage` is used only for first-use onboarding completion.
- Prompt presets are stored in browser cookies.
- No global client state library is used.

### Storage And Persistence

- Prisma ORM with SQLite.
- Database access is centralized through `src/lib/db.ts`.
- Core persisted models:
  - `Idea`
  - `FactoryPlan`
  - `Task`
  - `TaskBlocker`
- Local analytics are stored in `.creation-station/analytics-events.json` with retention capped at 500 events.
- Manual backup/restore is implemented in `src/lib/backup.ts` and the Settings UI.

### Important Components And Modules

- `src/app/page.tsx`: main workflow composition, idea filtering, review queue, recommendation panel.
- `src/app/components/TaskBoard.tsx`: task grouping, filtering, status transitions, blockers, staleness/waiting display.
- `src/app/components/FirstUseOnboarding.tsx`: first-use modal.
- `src/app/components/AppSidebar.tsx`: navigation and backup entry point.
- `src/app/settings/AnalyticsOverview.tsx`: local analytics summary.
- `src/app/settings/BackupRestore.tsx`: manual backup/restore UI.
- `src/lib/intelligence.ts`: current heuristic intelligence functions.
- `src/lib/pipelines.ts`: modular pipeline definitions and Factory guidance.
- `src/lib/task-labels.ts`: task label vocabulary and parsing.
- `src/lib/factoryPrompt.ts`: Factory prompt construction with detected pipeline context.
- `src/lib/aiProvider.ts`: isolated local Ollama integration.

### Existing Task, Idea, And Project Systems

- Ideas are the top-level intake records.
- Factory plans currently function as projects in the product language and backup format.
- Tasks belong to Factory plans.
- Task status is string-based and currently supports `TODO`, `DOING`, `BLOCKED`, `BACKLOG`, `DONE`, and `ARCHIVED`.
- Idea workflow status is enum-based in Prisma and supports intake, review, revision, approval, tasking, production, publishing, and archive states.
- Task dependencies are persisted through `TaskBlocker`.

## v1.6 Goals

1. Stabilize the existing heuristic intelligence layer as a clearly bounded foundation.
2. Make recommendations explainable enough that users understand why they are shown.
3. Improve next-best-work guidance using existing local data only.
4. Add lightweight insight surfaces without creating a new product subsystem.
5. Keep intelligence deterministic, testable, and local-first.
6. Preserve v1.5 workflow behavior and data safety.
7. Prepare a foundation that future AI features can use without adding agents, sync, accounts, or connectors.

## Non-Goals

- No autonomous agents.
- No agent meetings, runners, background automation, or delegated task execution.
- No external connectors.
- No cloud sync, accounts, authentication, teams, permissions, or multi-device conflict resolution.
- No full asset vault.
- No plugin system.
- No major Prisma redesign.
- No vector database, embeddings store, or semantic search infrastructure in v1.6.
- No external analytics services.
- No replacement of the current AI Factory Planner.

## Proposed Architecture

v1.6 should formalize intelligence as a read-only analysis layer over existing local records.

The preferred shape is:

```text
Prisma data -> intelligence input adapters -> scoring/rules -> recommendations/insights -> UI surfaces
```

The layer should stay modular but small:

- Keep database reads in pages or server helpers.
- Keep scoring and rule logic in `src/lib/intelligence.ts` or small adjacent files.
- Keep UI rendering in existing pages/components.
- Keep writes limited to existing user actions unless a specific insight preference is later approved.
- Use deterministic TypeScript functions that are easy to unit-test once a test script exists.

The first implementation should avoid schema changes if possible. Derived insight fields can be computed from existing `Idea`, `FactoryPlan`, `Task`, `TaskBlocker`, analytics events, timestamps, priorities, statuses, labels, routes, and pipeline definitions.

### Possible New Files

Only add these if they reduce complexity:

- `src/lib/intelligence-types.ts`

## Data Model Proposal

### Preferred v1.6 Approach: No Schema Change

Use derived local signals from existing data:

- Idea signals:
  - age
  - status
  - route/pipeline confidence
  - revision state
  - conversion state
  - plan/task counts

- Plan signals:
  - review waiting status
  - revision requested status
  - approved/task creation state
  - required asset count
  - risk text presence

- Task signals:
  - status
  - priority
  - labels
  - blocked state
  - dependency impact
  - staleness
  - project momentum

- Analytics signals:
  - task creation/completion counts
  - idea conversion count
  - backup/export/import activity
  - onboarding state

### Optional Future Schema, Only If Clearly Needed

If implementation proves that user-dismissed or pinned insights are necessary, propose a small table before coding:

```prisma
model IntelligencePreference {
  id        String   @id @default(cuid())
  key       String   @unique
  value     String
  createdAt DateTime @default(now())
  updatedAt DateTime @updatedAt
}
```

Do not add this in v1.6 unless a concrete UX need appears. The default plan is derived, stateless intelligence.

## Implementation Phases

### Phase 1: Baseline Audit And Test Fixtures

- Audit current `src/lib/intelligence.ts` behavior.
- Capture representative local fixture shapes for ideas, plans, tasks, blockers, and analytics.
- Identify unstable assumptions, duplicate scoring logic, and edge cases.
- No user-facing behavior change unless a clear bug is found.

### Phase 2: Intelligence Contract

- Define a stable recommendation shape:
  - `id`
  - `title`
  - `body`
  - `href`
  - `tone`
  - `priority`
  - `reasonCodes`
  - optional `confidence`
- Ensure all recommendation sources use the same scoring and dedupe path.
- Keep the current recommendation limit behavior.

### Phase 3: Rule And Score Refinement

- Improve prioritization for:
  - waiting review
  - revision loop completion
  - blocked work without details
  - cleared blockers
  - stale active work
  - next best task
  - high-confidence raw idea routing
- Keep rules deterministic and explainable.
- Avoid adding AI calls to recommendation generation.

### Phase 4: UI Polish For Explainability

- Update recommendation cards to show concise reasons or signal labels.
- Keep actions linked to existing surfaces: Review Inbox, Factory, Task Board, Idea Inbox.
- Avoid a new full-page intelligence center unless existing surfaces cannot support the information.

### Phase 5: Dashboard Summary

- Add a compact summary of intelligence health if useful:
  - open recommendations
  - blocked tasks
  - stale tasks
  - review/revision pressure
  - next task candidate
- Keep it read-only.

### Phase 6: QA And Release Notes

- Run lint and build.
- Verify the core workflow still works.
- Verify recommendations appear, disappear, and reprioritize after normal user actions.
- Document limitations and v2 deferrals.

## Validation Checklist

### Static Checks

- `npm run lint`
- `npm run build`

There is no current `test` or `typecheck` script in `package.json`.

### Manual Workflow Checks

- Create a new idea.
- Confirm idea routing signal appears where expected.
- Convert an idea through Factory.
- Confirm review recommendation appears for pending plans.
- Request revision.
- Confirm revision recommendation appears.
- Approve a plan.
- Confirm tasks are created.
- Move a task to `BLOCKED`.
- Add and clear a blocker.
- Confirm blocked/cleared recommendations update.
- Move a task to `DONE`.
- Confirm next-task and completion-related signals update.
- Archive a task.
- Confirm archived tasks do not dominate recommendations.
- Reload and confirm derived intelligence remains consistent.

### Data Safety Checks

- Confirm no existing tasks, ideas, plans, or blockers are deleted.
- Confirm no schema migration is required unless explicitly approved.
- Confirm manual backup export still includes all existing records.
- Confirm restore validation is not weakened.

## Rollback Strategy

1. Keep v1.6 implementation commits small and phase-based.
2. Prefer pure functions and UI-only consumption of derived signals.
3. If recommendation quality regresses, revert the intelligence scoring commit without touching persistence.
4. If UI clutter regresses workflow clarity, revert the UI integration commit while keeping pure helpers.
5. If any schema change becomes necessary, stop before implementation and create a separate migration plan with backup/restore validation.
6. Stable rollback baseline is tag `v1.5.0`.

## Risks

- Recommendation noise could reduce trust if signals are too eager.
- Stale-task logic can feel wrong if local timestamps shift during backup/restore.
- Existing `src/lib/intelligence.ts` can become too large if new rules are added without separating types or summaries.
- The term "AI Recommendations" may overpromise because current recommendations are deterministic heuristics.
- Adding explainability UI could clutter the already dense home page.
- Analytics are local file writes and not suitable for high-concurrency signal generation.
- v1.6 scope could drift into agents or automation unless non-goals stay explicit.

## Recommended First Implementation Slice

Start with a no-schema refactor of the intelligence recommendation contract:

1. Add reason codes and priority/confidence to the current recommendation type.
2. Preserve current recommendation behavior.
3. Update the existing recommendation panel to show one concise reason per card.
4. Validate lint/build and the core workflow.

This is the smallest useful step because it improves trust and testability without new architecture.
